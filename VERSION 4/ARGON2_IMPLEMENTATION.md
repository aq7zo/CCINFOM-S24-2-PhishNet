# Argon2 Implementation Overview

This document provides a general overview of how Argon2 password hashing has been implemented in the PhishNet Cybersecurity Incident Reporting System.

## Introduction

PhishNet uses **Argon2id** for secure password hashing, which is currently considered the best practice for password storage. Argon2 won the Password Hashing Competition in 2015 and has become the recommended choice for modern applications.

## Why Argon2?

Argon2 was chosen over older hashing methods (like SHA-256 or MD5) because:

- **Security:** Resistant to GPU-based attacks, side-channel attacks, and memory-time trade-off attacks
- **Adaptability:** Can be tuned to balance security and performance based on hardware capabilities
- **Industry Standard:** Recommended by security experts and used by major organizations
- **Future-Proof:** Designed to remain secure as computing power increases

## Implementation Architecture

### Core Component: SecurityUtils

The Argon2 implementation is centralized in the `SecurityUtils` class, which serves as the single source of truth for all password-related security operations. This design ensures:

- **Consistency:** All password operations use the same configuration
- **Maintainability:** Security settings can be updated in one place
- **Reliability:** Prevents configuration mismatches between hashing and verification

### Library Used

The implementation uses the **password4j** library (version 1.7.3), which provides a clean Java API for Argon2 operations. This library:

- Handles the low-level Argon2 operations
- Manages salt generation automatically
- Provides a simple interface for hashing and verification
- Supports multiple Argon2 variants (id, i, d)

## Argon2 Configuration

The system uses carefully chosen parameters that balance security and performance:

### Parameters

- **Variant:** Argon2id (hybrid of Argon2i and Argon2d)
- **Memory Cost:** 65,536 KiB (64 MiB)
- **Time Cost (Iterations):** 3
- **Parallelism:** 4 threads
- **Salt Length:** 16 bytes (128 bits)
- **Hash Length:** 32 bytes (256 bits)

### Why These Parameters?

- **Memory Cost (64 MiB):** High enough to resist GPU attacks, low enough for reasonable performance
- **Iterations (3):** Provides good security without excessive delay
- **Parallelism (4):** Utilizes modern multi-core CPUs effectively
- **Argon2id Variant:** Best balance between resistance to side-channel attacks and GPU attacks

## How It Works

### Password Hashing Flow

1. **User Registration:**
   - User enters a plaintext password
   - System validates password (length, format, etc.)
   - `SecurityUtils.hashPassword()` is called
   - Argon2id generates a random salt
   - Password is hashed with salt and configured parameters
   - Complete hash string (including parameters and salt) is stored in database

2. **Password Storage Format:**
   ```
   $argon2id$v=19$m=65536,t=3,p=4$[salt]$[hash]
   ```
   - The hash is self-contained: includes variant, version, parameters, salt, and hash
   - This allows verification without storing parameters separately

### Password Verification Flow

1. **User Login:**
   - User enters email and password
   - System retrieves stored hash from database
   - `SecurityUtils.verifyPassword()` is called with plaintext password and stored hash
   - System extracts parameters from stored hash
   - Password is hashed again with same parameters and salt
   - Hashes are compared (using constant-time comparison)
   - Login succeeds if hashes match

2. **Critical Implementation Detail:**
   - The same `Argon2Function` instance must be used for both hashing and verification
   - This ensures parameter consistency and reliable verification

## Security Features

### Salt Generation

- Each password gets a unique, random salt
- Salt is automatically generated by the password4j library
- Salt is embedded in the hash string
- Even identical passwords produce different hashes

### Constant-Time Comparison

- Password verification uses constant-time string comparison
- Prevents timing attacks that could reveal password information
- Comparison always takes the same time regardless of where differences occur

### Input Validation

- Password length limits (max 128 characters) prevent DoS attacks
- Empty passwords are rejected
- Null values are handled safely

### Legacy Support

- System supports migration from older SHA-256 hashes
- Legacy hash verification is handled transparently
- Allows gradual migration without breaking existing accounts

## Integration Points

### User Registration

- `SignUpController` → `VictimAuthenticationService` → `SecurityUtils.hashPassword()`
- Password is hashed before being stored in database
- Hash is stored in `Victims` table `PasswordHash` column

### User Login

- `LoginController` → `VictimAuthenticationService` → `SecurityUtils.verifyPassword()`
- Stored hash is retrieved from database
- Plaintext password is verified against stored hash

### Admin Authentication

- `AdminLoginController` → `AdminAuthenticationService` → `SecurityUtils.verifyPassword()`
- Same security standards apply to admin accounts
- Admin passwords also use Argon2id

## Database Schema

### Storage

- Password hashes are stored in `VARCHAR(255)` columns
- Format: `$argon2id$v=19$m=65536,t=3,p=4$[salt]$[hash]`
- Typical hash length: ~100-150 characters
- Original passwords are never stored

### Tables Using Argon2

- **Victims:** Public user accounts
- **Administrators:** Admin and staff accounts

## Performance Considerations

### Hashing Speed

- Typical hash generation: 100-500ms per password
- Acceptable for user registration (one-time operation)
- Fast enough for good user experience

### Verification Speed

- Password verification: Similar speed to hashing
- Fast enough for login operations
- No noticeable delay for users

### Resource Usage

- Memory: ~64 MiB per hash operation (configurable)
- CPU: Multi-threaded (4 threads) for efficiency
- Scalable: Parameters can be adjusted for different hardware

## Migration Strategy

### From Legacy Hashes

The system supports a migration period where both Argon2 and SHA-256 hashes are accepted:

1. Old accounts continue to work with SHA-256 hashes
2. New accounts use Argon2id
3. When users log in, system detects hash type
4. Legacy verification is handled separately
5. Eventually, all accounts can be migrated to Argon2

### Migration Process

- Users can be migrated automatically on next login
- Or manually updated in database
- Legacy support can be removed after migration completes

## Best Practices Followed

1. **Centralized Security:** All password operations go through SecurityUtils
2. **Consistent Configuration:** Same parameters for all operations
3. **Proper Salt Usage:** Unique salt per password
4. **Secure Comparison:** Constant-time comparison prevents timing attacks
5. **Input Validation:** Prevents DoS and invalid inputs
6. **Error Handling:** Graceful failure without exposing information
7. **Self-Contained Hashes:** Parameters embedded in hash string

## Security Benefits

### Resistance to Attacks

- **Brute Force:** High memory cost makes attacks expensive
- **GPU Attacks:** Memory-hard function resists GPU acceleration
- **Side-Channel Attacks:** Argon2id variant provides protection
- **Rainbow Tables:** Unique salts prevent precomputed attacks
- **Timing Attacks:** Constant-time comparison prevents information leakage

### Compliance

- Follows industry best practices
- Meets security standards for sensitive applications
- Suitable for compliance requirements (e.g., data protection regulations)

## Future Considerations

### Parameter Tuning

- Parameters can be adjusted based on:
  - Hardware capabilities
  - Security requirements
  - Performance needs
- Should be reviewed periodically as computing power increases

### Algorithm Updates

- Argon2 remains current, but future algorithms may emerge
- Architecture allows for easy replacement if needed
- Centralized implementation simplifies updates

## Conclusion

The Argon2 implementation in PhishNet provides:

- **Strong Security:** Industry-leading password hashing
- **Good Performance:** Balanced parameters for user experience
- **Maintainability:** Centralized, well-documented code
- **Flexibility:** Supports migration and future updates
- **Reliability:** Consistent, tested implementation

This implementation follows security best practices and provides a solid foundation for protecting user credentials in the PhishNet system.
